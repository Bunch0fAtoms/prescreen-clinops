Masking without breaking downstream

A short field guide to how a column mask ripples into the joins and models other teams build. It explains what happened on Day 1, the trade-offs of each masking style, and how to classify protected data at scale without a separate function for every data type.

The one thing to remember

A mask applies everywhere the column is read, including join keys. Masking a shared join key like person_id to NULL silently breaks every join for anyone the policy covers. Choose the mask style by what downstream needs, not just by what to hide.

01 What happened on Day 1

The mask worked exactly as designed

The governance team did the right things in the right order. They classified protected health information (PHI), tagged the columns, and bound a column mask that hides identifiers from the restricted research group. The behavior that stopped the ML team was the mask doing its job on a column that the whole pipeline depends on.

The kit's mask on the patient identifier returns NULL for members of the restricted group:

-- kit: notebooks/02_column_masks.py
CREATE OR REPLACE FUNCTION mask_person_id(pid BIGINT)
  RETURN CASE WHEN is_account_group_member('<OCDO_GROUP>')
              THEN NULL ELSE pid END;

ALTER TABLE person ALTER COLUMN person_id SET MASK mask_person_id;

person_id is the foreign key that joins all six OMOP tables (person, condition_occurrence, measurement, observation, drug_exposure, note). Once the ML identity fell under the policy, every person_id it read came back NULL. A join on NULL matches no rows, so the star-schema joins collapsed and the models had nothing to work with. Nothing errored loudly. The rows just quietly went to zero.

Why it surprised the room

The mask is scoped to a column, and it fires for every reader the policy applies to, including the service identity running a scheduled job. So a governance change on the shared source reached into a pipeline that no one edited. This is expected Unity Catalog (UC) behavior, not a defect. The lesson is that masking a join key is a downstream event, and it needs to be sequenced and communicated like one.

02 Masking styles and their cost

Each style hides the value differently, and each has a different downstream cost

A column mask is just a function that rewrites the value before a restricted user sees it. What you return decides whether joins, group-by, and model features still work. Here are the common styles, with what each does to the data other teams build on.

Conditional

The first three hide the value and lose the join. The last two are the tools that let you protect data and keep it usable at the same time. The Day 1 mask was the nullify style on a join key, which is the one combination that stops everything downstream.

03 The rule for join keys

Never nullify a key that another team joins on. Tokenize it instead.

A patient identifier used as a join key needs to survive masking. Two patterns do that. Pick by whether the pipeline is allowed to see the real identifier.

Pattern A · deterministic token (recommended for keys)

Return a stable stand-in instead of NULL. The same real id always maps to the same token, so joins line up on both sides, and the real id is never revealed. The token stays a BIGINT, so it still matches the column type.

CREATE OR REPLACE FUNCTION mask_person_id(pid BIGINT)
  RETURN CASE WHEN is_account_group_member('<OCDO_GROUP>')
              THEN xxhash64(CAST(pid AS STRING))  -- stable BIGINT token
              ELSE pid END;

Now a restricted researcher joins person to measurement on the token and gets the right rows, without ever seeing a real patient id. Privacy and joinability at once.

Pattern B · except the pipeline identity

If the scheduled model job is allowed to read real identifiers, keep its service principal out of the restricted group, or list it in the policy's EXCEPT clause. Human researchers still see masked values. The pipeline reads raw and joins normally. Use this only when the pipeline is cleared to see the real id.

Pattern A is usually the better default here, because it holds even when you are not sure which identities will read the data later. It fails safe: joins keep working and nobody sees a real identifier.

04 Why one tag needed many functions

A mask function must return the column's type, so one tag can span several functions

The governance team tagged the columns and then wanted one policy on the tag to mask them all. The friction they hit is real and expected. A mask function's return type must be castable to the type of the column it masks. Columns with the same tag can have different types, so one function cannot always cover them all.

In the kit, the phi = other_identifier tag sits on both a BIGINT column and a STRING column. That forces two functions:

-- BIGINT identifier
CREATE FUNCTION mask_person_id(pid BIGINT) RETURN ... ;      -- returns BIGINT
-- STRING free-text identifier
CREATE FUNCTION mask_note_text(txt STRING) RETURN ... ;      -- returns STRING

An attribute-based access control (ABAC) policy binds one mask function to a tag. Because mask_person_id returns a BIGINT, a policy using it only fits the BIGINT columns that carry the tag. The STRING column with the same tag needs its own policy with the STRING function. So the count of policies follows the count of data types, not the count of tags.

-- one policy per type, matched by the shared tag
CREATE POLICY redact_id_bigint  ON CATALOG <CATALOG>
  COLUMN MASK mask_person_id  TO `<OCDO_GROUP>`  FOR TABLES
  MATCH COLUMNS hasTagValue('phi','other_identifier') AS c  ON COLUMN c;

CREATE POLICY redact_id_string ON CATALOG <CATALOG>
  COLUMN MASK mask_note_text  TO `<OCDO_GROUP>`  FOR TABLES
  MATCH COLUMNS hasTagValue('phi','other_identifier') AS c  ON COLUMN c;

The scaling fix · a small typed function library

You do not need a function per column. You need one per type family, written once and reused across the whole catalog:

  • One STRING function for names, notes, and text identifiers.
  • One VARIANT function for the numeric family. A mask that accepts and returns VARIANT covers INT, BIGINT, DECIMAL, and DOUBLE in a single function, because those cast cleanly through VARIANT.
  • One function each for DATE and TIMESTAMP where you tag those.

Classify once, and the tag policy travels with the tag to every new table the DE and ML tracks create. That is the scale the team was reaching for. The typed function library is the piece that makes it click.

Worked example: two tags, four types, one function library

Two governed tags, phi = other_identifier and phi = quasi_identifier. Each lands on a BIGINT column and a numeric column, and the second tag also carries a DATE and a TIMESTAMP so every function is in play. First the library, written once:

-- 1. BIGINT join keys -> stable token: joins still work, real id never shown
CREATE OR REPLACE FUNCTION mask_id_bigint(id BIGINT)
  RETURN xxhash64(CAST(id AS STRING));
-- 2. any numeric (INT, BIGINT, DECIMAL, DOUBLE) -> redacted, via VARIANT
CREATE OR REPLACE FUNCTION mask_num_variant(v VARIANT)
  RETURN CAST(NULL AS VARIANT);
-- 3. DATE -> generalized to year only
CREATE OR REPLACE FUNCTION mask_date(d DATE)
  RETURN make_date(year(d), 1, 1);
-- 4. TIMESTAMP -> truncated to the day
CREATE OR REPLACE FUNCTION mask_ts(ts TIMESTAMP)
  RETURN date_trunc('DAY', ts);

Classify the columns, then bind one policy per tag and type. Every policy reuses a library function, nothing is re-authored:

-- tag the columns
ALTER TABLE patient ALTER COLUMN patient_id          SET TAGS ('phi'='other_identifier'); -- BIGINT
ALTER TABLE patient ALTER COLUMN insurance_member_no SET TAGS ('phi'='other_identifier'); -- DECIMAL
ALTER TABLE patient ALTER COLUMN provider_id         SET TAGS ('phi'='quasi_identifier'); -- BIGINT
ALTER TABLE patient ALTER COLUMN household_income    SET TAGS ('phi'='quasi_identifier'); -- DOUBLE
ALTER TABLE patient ALTER COLUMN date_of_birth       SET TAGS ('phi'='quasi_identifier'); -- DATE
ALTER TABLE patient ALTER COLUMN last_visit_ts       SET TAGS ('phi'='quasi_identifier'); -- TIMESTAMP

-- one policy per (tag, type), all reusing the library

CREATE POLICY mask_a_nums ON SCHEMA clin.omop COLUMN MASK mask_num_variant TO ocdo_researchers
  FOR TABLES MATCH COLUMNS has_tag_value('phi','other_identifier') AS c ON COLUMN c;

  FOR TABLES MATCH COLUMNS has_tag_value('phi','quasi_identifier') AS c ON COLUMN c;

One gotcha to plan for

VARIANT is a superset of the numeric types, so mask_num_variant is also type-compatible with your BIGINT keys. That means patient_id matches both the tokenizer and the redactor. A column takes at most one mask, resolved by policy precedence, so give the BIGINT tokenizer policies the higher precedence, or a key you meant to keep joinable gets nulled, which is the Day 1 failure again. For zero ambiguity, give join keys their own tag value, for example phi = joinable_id, so the tokenizer matches only keys and the redactor never touches them.

Note: tag-based ABAC policies are generally available in Unity Catalog, but may be feature-gated on a given workspace. The kit wraps the CREATE POLICY step in a try/except so the build stays green, and falls back to the per-table masks, which already deliver the same protection. The policy pattern is what to adopt as the workspace enables it.

05 A safe sequence for Day 2

Classify, protect, and prove, in an order that does not stall the other tracks

The same governance goal, sequenced so downstream teams keep moving. Five steps.

  1. Decide per column: does anyone downstream join or read it? If it is an identifier used as a key, plan a deterministic token, not NULL. If it is free text to hide fully, redact is fine.
  2. Separate people from pipelines. Keep DE and ML service principals out of the restricted group, or list them in EXCEPT, so scheduled jobs read what they need while human researchers see masked values.
  3. Apply, then prove with a downstream join. After binding a mask, run one real join as the restricted identity and confirm rows still flow. The kit's verification cells and information_schema.column_masks show the mask is bound; a join proves it did not break the pipeline.
  4. Announce changes to the shared source. Masks and filters inherit to every reader. A one-line heads-up before a policy lands on shared tables saves an afternoon of silent zero-row results.
  5. Move to tag policies with the typed function library. Classify once, and new tables inherit masking automatically. This is the pattern to carry past the workshop.
