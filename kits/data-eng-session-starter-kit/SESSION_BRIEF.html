3 How the build works

The wiring is done. You build the logic.

This is a starter kit, not a finished solution. The boilerplate is wired so your time goes to the engineering that teaches the platform. Look for the # TODO (you build this) markers.

✅ Pre-built for you
  • Six synthetic OMOP source tables, read-only. You ingest from them.
  • The bronze landing writers and the "new column" version-2 frame.
  • An injected reconciliation gap, and the live trials feed with its Auto Loader ingest to a VARIANT bronze.
🛠️ You build the core
  • The mergeSchema append, the count comparison, and the missing-key anti-join.
  • The assert_ingest_allowed() gate and the in_sla_window() guard.
  • The quarantine that routes bad trial records aside with a reason, so a malformed line never fails the load.

Config over code

The ingest gate and the SLA window are driven by Unity Catalog config, not hardcoded. A steward changes behavior without a deploy, and every decision is auditable.

4 Your layer feeds the ML group

You own the curated structured layer the pre-screen builds on

This is the contract that lets both groups run in parallel.

The decoupling

You produce the structured silver the Applied AI group consumes

The structured silver tables — silver_biomarker_profile, silver_demographics, silver_prior_therapy — are produced by your pipeline. So the Applied AI group is never blocked, that layer is pre-seeded as a reference on Day 1. They start from it while you rebuild and harden the pipeline that produces it. Your success bar includes parity: your output matches the reference table names and counts, about 240 / 300 / 146.

Day-2 highlight, never a blocker

If you finish ahead, point the ML group at your freshly-built silver, a live "DE → ML swap." The pre-seed is always there, so this is upside, never a dependency.

5 Ground rules

Governed, scoped, synthetic

A security-first customer. These rules are the same across all four sessions.

Read-only source

The six OMOP tables and the trials Volume are shared and read-only. Everything you create lands in your own schema.

Governance applies to your tables

The Governance group's tag → mask → row-filter pattern re-applies to the silver you build. Reliable ingestion and governed access are one story.

Unity-Catalog-scoped

Catalog and schema come from bundle variables. No hive_metastore, ever. No hardcoded secrets.

Synthetic data only

No real PHI. The same logic runs against real curated_omop when Fred Hutch is ready.
